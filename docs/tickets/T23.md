## Goal

Formally record the B-b feasibility outcome for CICIoT2023 on the available raw CSV artifact, encode the outcome as an enum, write the rejection report, and prevent any downstream B-b training from running.

## Why This Exists

Phase 0 GA-07 audit found processed files contained only 39 feature columns. The H02 verification (2026-05-24) confirmed the raw CSV artifact under `data/raw/CIC_IOT_Dataset2023/CSV/` is structurally identical: 39 numeric flow features + `Label` (merged only), with no MAC, device, client, source/destination IP, capture-source identifier, or timestamp column. CICIoT2023 B-b is therefore infeasible on the currently available CSV artifact. This ticket implements that documented infeasibility outcome as a first-class artifact.

## Scope

- Encode the outcome `B_B_REJECTED_NO_METADATA` as an enum value alongside any other allowed B-b feasibility outcomes.
- Record the schema evidence (column list, sampled file paths, sampled file row counts, and the explicit absence of MAC / device / client / capture-source / source-destination IP / timestamp columns) in a rejection artifact / report under the canonical outputs path.
- Write a rejection manifest that downstream code can read so that B-b training/scoring is structurally prevented from running on this CSV artifact.
- Add a guard so that any later attempt to start B-b training on CICIoT2023 fails fast with the recorded reason.
- Do not define any client partition. Do not fabricate pseudo-clients from folders, merged files, labels, row chunks, or filenames.

## Out of Scope

- Training on B-b data (that is T24)
- Modifying existing Regime B-a processing
- Changing the 39-feature selection
